# 메모리 · 컨텍스트 연속성 (Memory & Context)

> **파일명 유래**: `memory`(기억) + `context`(맥락). 세션이 끊겨도 기억을 파일로 남겨 맥락을 잇는 규칙이라 이 이름이다.

> 에이전트는 세션 간 기억이 없다. 이 문서는 **프로젝트 내 지식 재사용**을 위한 패턴을 정의한다.
> 이미지 07 컨텍스트 · 12 메모리 개념 반영.
> 최종 검토일: 2026-06-18.

---

## 0. 세션 간 기억 = 파일로 남긴다

> **[MUST] 기억은 머리가 아니라 파일에 남긴다.** 에이전트는 매 세션 기억이 0으로 초기화된다. 파일에 안 남긴 결정·진행·맥락은 다음 세션에 **존재하지 않는 것과 같다.** "기억하겠지"는 없다.

에이전트는 매 세션을 새로 시작한다. 연속성은 **파일**로 만든다:

| 기억 유형 | 저장 위치 | 형식 |
|-----------|-----------|------|
| 결정 이유 | `docs/specs/decisions/ADR-*.md` | ADR |
| 작업 기록 | `docs/plans/PLAN-*.md` | PRD + 상태 |
| 작업 인덱스 | `docs/plans/INDEX.md` | 전체 PLAN 한눈에(상태·날짜). 100개+ 일 때 필수 |
| 완료 보관 | `docs/plans/archive/` | Done 되고 참조 끝난 PLAN을 옮겨 보관 |
| 변경 이력 | `docs/plans/CHANGELOG.md` | 날짜·내용·이유 |
| 미룬 것 | `docs/plans/BACKLOG-deferred.md` | 표 |
| 세션 인수인계 | `docs/plans/HANDOFF.md` (선택) | 자유 형식 |

> **세션 길이 관리 (긴 세션에서 새는 것 방지)**: 세션이 길어지면 앞부분 맥락이 흐려져 가끔 기존 대화를 무시하고 딴 길로 샌다. 작업 단위가 끝나면 `/compact`(대화 요약·압축) 또는 `/clear`(초기화)로 세션을 정리한다. 맥락은 위 파일들(roadmap·PLAN·ADR)이 잡아주므로, 세션을 끊어도 새 세션이 복원한다. *세션을 안 끊고 무한정 이어가지 않는 게 가장 확실한 예방*이다.

## 0-1. 기억은 성격별로 분리한다 (안 그러면 오염된다)

> 장기 운영 에이전트의 기억 오염 패턴: 항구적 사실과 일회성 로그가 같은 통에 섞이면, 중요한 정보를 찾기 어려워지고 토큰이 낭비되며 잘못된 의사결정으로 이어진다. 답은 나중에 청소하는 게 아니라 **처음부터 안 섞는 것**이다.

| 층 | 성격 | 어디에 | 예 |
|----|------|--------|-----|
| 항구층 | 거의 안 변함 | PRD·ADR·CLAUDE.md | 스택 선택, 핵심 결정, 보안 프로파일 |
| 상태층 | 자주 변함 | PLAN·INDEX 상단·HANDOFF | 진행 중 작업, 현재 위치, 다음 할 일 |
| 휘발층 | 그 순간만 유효 | **기억 파일에 남기지 않는다** | 빌드 로그, 에러 메시지 원문, 임시 포트, 파일 개수 |

- [MUST] 휘발층(로그·에러 원문·임시값)을 PLAN·INDEX·CHANGELOG에 붙여넣지 않는다. 필요하면 "요지 한 줄"로 변환해 남긴다(예: "500 에러 원인=환경변수 누락"은 상태층, 스택트레이스 전문은 휘발층).
- Done된 PLAN은 상태층에서 빨리 분리한다(archive/): 참조 끝났으면 이동(doc-sync의 이동 제안 트리거). 끝난 것이 현재 뷰에 계속 남는 것 자체가 오염이다.
- 같은 사실이 두 곳에 있으면 한 곳을 지운다(항구층 우선). 상충하면 사람에게 묻는다.

## 1. 세션 시작 체크리스트 (에이전트용)

> Claude Code에선 session-start 훅이 로드맵 위치·진행 중 PLAN·인수인계를 세션 시작 시 자동 주입한다(기계화). 아래 체크리스트는 그 보완이자, 훅이 없는 도구에서의 폴백이다.

> **[MUST] 맥락 복원 없이 작업 시작 금지.** 새 세션은 기억이 0이다. 아래를 건너뛰고 바로 작업하면 ─ 이미 한 결정을 또 하고(ADR 무시), 끝난 작업을 다시 하고(PLAN 무시), 로드맵에서 벗어난다. **복원은 선택이 아니라 착수 전제다.**
>
> **ROADMAP.md가 있으면 그 복원이 최우선이다.** 미팅·다른 업무로 컨텍스트 스위칭이 잦은 1인 작업에서, 세션이 끊겼다 돌아올 때 "마지막으로 어디까지 했고 다음이 무엇인지"를 ROADMAP "현재 위치"로 가장 먼저 맞춘다. 이 싱크가 안 맞은 채 작업하면 가장 크게 샌다.

새 세션에서 작업을 시작하기 전, **조용히** 아래 순서로 맥락을 복원한다:

```
1. CLAUDE.md 로드 확인 (/memory 또는 첫 응답에서)
2. ROADMAP.md 있으면 "현재 위치"부터 확인 (어느 단계·다음 진입점·미정 목록) → 큰 프로젝트
3. 관련 PLAN-*.md 상태 확인 (In Progress / Blocked / Done). PLAN이 많으면 `INDEX.md`로 전체 상태를 먼저 훑는다
4. 최근 CHANGELOG.md 3~5줄 확인
5. 관련 ADR 확인 (같은 결정을 또 하지 않기 위해)
6. BACKLOG에서 이번 작업과 연관된 미룬 항목 확인
```

- 이 체크리스트를 **장황하게 서술하지 않는다.** 복원한 **결과(파악한 맥락)만** 한 줄로 언급.
- 복원할 파일이 없으면(첫 작업) "복원할 이력 없음"이라고 명시하고 시작한다. **침묵하고 건너뛰지 않는다** ─ 건너뛴 건지 없는 건지 사람이 알아야 한다.
- 큰 프로젝트(ROADMAP 있음)에서 이 복원을 빠뜨리면 멈춰서 다시 한다. `/resume`이 이 복원을 강제 실행하는 진입점이다.

## 2. HANDOFF.md: 세션 인수인계 (선택)

작업이 여러 세션에 걸쳐 이어질 때, 또는 다른 에이전트/사람이 이어받을 때 사용.

```markdown
# 인수인계: <작업명> (<날짜>)

## 현재 상태
- PLAN: docs/plans/PLAN-<이름>.md (In Progress)
- 완료된 것: ...
- 남은 것: ...
- 막힌 지점 (있으면): ...

## 다음 세션 진입점
- 시작할 파일/단계: ...
- 주의사항: ...

## 열린 질문 (사람 결정 필요)
- ...
```

## 2-1. 여러 PC를 오가며 이어 작업하기 (멀티 디바이스)

> 여러 작업 환경(여러 PC·집/사무실·노트북 등)을 오가며 한 프로젝트를 이어갈 때. git이 "바톤"을 넘긴다 ─ 단, **git은 "커밋된 것"만 옮긴다.** 코드뿐 아니라 상태 파일(roadmap·PLAN·HANDOFF)도 커밋돼야 다음 환경이 이어받는다.

### 떠나는 환경에서 (세션 마무리)
1. **상태를 파일에 남긴다**: roadmap "현재 위치"·PLAN 상태 갱신. 디테일이 필요하면 `/handoff`로 HANDOFF.md 작성("방금 뭐 하던 중"까지).
2. **사용자가 commit + push** 한다("커밋하고 푸시해"). 코드 + 상태 파일이 함께 올라가야 한다.
3. **로컬 = 원격 일치 확인 후 세션 종료**: `git status`가 깨끗하고(미커밋 없음), 로컬 HEAD와 GitHub 커밋이 일치하는지 확인. 미커밋·미푸시가 있으면 다음 환경에서 안 보인다(v2.21 "로컬 ≠ 커밋"과 같은 함정).

### 이어받는 환경에서 (세션 시작)
1. **사용자가 git pull** 로 최신을 받는다(GitHub 커밋 확인 후).
2. **새 세션 시작 → 위치 복원**: AI가 §1 세션 시작 체크리스트대로 roadmap·PLAN·HANDOFF를 읽고 "어디까지 했고 다음은 무엇"을 복원한다(`/resume`). 이게 "바톤 받기"다.
3. 복원 결과를 한 줄 브리핑하고, 그 지점부터 이어갈지 확인받은 뒤 진행.

### 약한 고리 (알고 있을 것)
- **push를 빠뜨리면 바톤이 안 넘어간다.** 떠나기 전 commit+push가 필수. (사용자가 직접 챙기는 지점)
- **HANDOFF.md는 자동이 아니다.** `/handoff`를 호출해야 작성된다. 깜빡해도 roadmap·PLAN 상태로 *어느 정도는* 복원되지만, "방금 뭐 하던 중"의 디테일은 handoff가 있어야 산다.
- **여러 환경에서 동시에 작업하지 않는다.** 한 번에 한 곳에서만(순차). 동시는 멀티세션 충돌과 같은 문제(`multi-agent.md §0-1` 형태 B).

## 3. ADR 재사용: 같은 결정 반복 방지

에이전트가 기술 선택·설계 결정을 할 때:

1. `docs/specs/decisions/`에서 관련 ADR을 먼저 검색
2. 이미 결정된 사항이면 그대로 따름 (재논의 금지)
3. 기존 ADR과 충돌하는 방향이 필요하면 → **사람 승인 + 새 ADR** 작성 후 진행

ADR 번호 규칙: `ADR-NNN-<짧은-제목>.md` (NNN은 3자리 순번)

## 4. RAG 연동 (선택: 문서가 많아질 때)

`docs/` 내 문서가 많아져 에이전트가 관련 컨텍스트를 찾기 어려울 때:

```
옵션 A: /spec 또는 /feature 명령 시 관련 PLAN·ADR·CHANGELOG를 자동 include
         → CLAUDE.md §5에 "신규 작업 시 관련 과거 PLAN 첨부" 규칙 추가

옵션 B: 외부 RAG 도구 연동 (Notion, Obsidian, 자체 임베딩)
         → guardrails.md AI/LLM 보안 섹션 기준 적용 필수

옵션 C: HANDOFF.md를 매 작업 말미에 갱신 (세션 간 단일 브리지)
```

규모가 작을 때는 옵션 C만으로 충분하다. (YAGNI)
