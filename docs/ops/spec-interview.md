# 착수 전 명세 인터뷰 (Spec Interview): 막연한 목표 → 정밀 명세

> **파일명 유래**: `spec`(명세)을 `interview`(인터뷰)로 끌어낸다. 막연한 목표를 질문으로 캐물어 정밀 명세로 바꾸는 과정이라 이 이름이다.

> `agent-loop.md` ①정렬의 **[MUST] 막연하면 실행 금지**를 실행하는 절차.
> 목적: "어떤 거 만들고 싶어" 같은 막연한 한 줄을, 코드 짜기 전에 **검증 가능하고 완성도 높은** 명세로 빚는다.
> 구성: **§0 동기 + 11축** (총 12축) + 깊이 선택(빠른/표준/깊게) + 체크리스트·점수 이중 강제.

---

## 0. 시작: 깊이를 먼저 고른다

인터뷰를 시작하면 **맨 먼저 깊이를 묻는다**. (사용자가 미리 지정했으면 건너뜀)

```
이 작업의 인터뷰 깊이를 골라주세요:
  [빠른]  핵심 3축만: 사소·명확한 작업 (오타·문구·소규모 수정)
  [표준]  7~8축: 일반 기능 (대부분 여기)
  [깊게]  11축 전부: 제품·서비스·고위험 (돈·인증·데이터·외부영향)
```

깊이는 **묻는 축의 수**를 정한다. 깊을수록 완성도는 오르고 질문은 늘어난다.
사용자가 도중에 "그만/네가 정해"라고 하면 즉시 멈추고 남은 축은 가정으로 명시한다.

> 처음이거나 부담되면 **[빠른]이나 [표준]으로 시작**해도 된다. 깊이는 언제든 올릴 수 있고,
> 답하기 어려운 질문은 "모름"이라고 답하면 가정으로 채우고 넘어간다. 막혀서 멈출 일은 없다.

---

## 1. 규율 (깊이 무관 공통)

- **[MUST] 소크라테스식 1문 1답**: 질문을 한꺼번에 쏟지 않는다. **한 번에 하나씩** 묻고, 이전 답을 받아 다음 질문을 빚는다. 답이 모호하면 더 좁혀 되묻는다.
- **[MUST] 진행 표시**: 매 질문 앞에 반드시 `[축 N/총축수]`를 붙인다 (예: `0축 동기 (1/12)`, `★1축 진짜 목표 (2/12)`). 사용자가 지금 어디쯤인지, 몇 개 남았는지 항상 알게 한다.
- **[MUST] 끝까지 묻기 (조기 개발 금지)**: 고른 깊이의 **모든 축을 다 묻기 전에는 절대 코드·파일 생성을 시작하지 않는다.** "충분히 모인 것 같다"는 판단으로 중간에 개발로 넘어가는 것은 금지. 마지막 축까지 묻고 → PRD 작성 → "이대로 맞나" 확인 → *그 다음에야* 실행. 한 축이라도 건너뛰려면 사용자에게 명시적으로 "이 축 건너뛸까요?"라고 묻고 동의를 받는다.
- **[MUST] 각 축 답변 후 요약 반영**: 답을 받으면 한 줄로 "지금까지 정리: ..."를 갱신해 보여준다. 사용자가 흐름을 놓치지 않고, 틀린 걸 즉시 바로잡게 한다.
- **[MUST] 되돌리기 허용**: 사용자가 "앞 질문 다시 / 아까 그건 틀렸어 / N축으로 돌아가자"고 하면 즉시 그 축으로 돌아가 답을 고친다. 인터뷰는 일방통행이 아니다.
- **[MUST] 조사 먼저**: 코드·로그·문서로 알 수 있는 축은 묻지 말고 채운다. *진행을 막는 미지수만* 질문한다(되묻기 피로 방지). 단, 조사로 채운 축도 "이 축은 레포에서 확인해 ...로 채웠습니다"라고 **밝히고 넘어간다**(조용히 건너뛰지 않는다).
- **[MUST] 탈출구**: 사용자가 "그만 / 네가 정해 / 기본값으로"라고 하면 즉시 중단하고, 남은 축은 **가정으로 명시**한 뒤 진행한다. (이때만 끝까지-묻기 규칙이 면제된다 = 사용자가 명시적으로 중단을 택한 경우.)
- **[MUST] 합의 게이트**: 인터뷰 결과를 PRD 초안으로 적고 **"이대로 맞나" 확인을 받은 뒤** 실행한다(작업=`docs/plans/`, 제품=`docs/specs/`).
- **[MUST] 깊이 존중**: 고른 깊이의 축만 묻는다. 빠른을 골랐는데 11축을 다 묻지 않는다(피로). 단, 답변에서 고위험 신호(돈·인증·삭제·외부전송)가 나오면 **깊이 상향을 1회 제안**한다.
- **[MUST] 선택형이 기본 (AskUserQuestion 활용)**: 모든 질문은 **번호 선택형**으로 제시하는 것을 기본으로 한다. 가능한 답을 2~4개 보기로 만들고, **항상 마지막 보기로 "직접 입력(기타)"을 둔다** (Claude Code의 선택 프롬프트는 자유 입력 옵션을 지원). 사용자가 ↑↓로 고르고 Enter로 답하게 한다. 자유서술만 요구하는 열린 질문은 *예외적으로만* 쓴다.
  - **보기 만드는 법**: 그 축에서 흔하거나 가능성 높은 답 2~4개를 구체적으로 제시한다 (막연한 "예/아니오" 말고). 예: "데이터 저장 위치?" → 1.로컬파일 2.DB 3.외부서비스(BaaS) 4.저장 안 함 5.직접 입력.
  - **열린 질문 예외**: 진짜 목표·가치처럼 *생각 자체를 끌어내야* 하는 축은, 먼저 가능한 방향을 보기로 제시하되 "직접 입력"을 눈에 띄게 두어 사용자가 자기 말로 쓰도록 유도한다. 보기로 사고를 가두지 않는다.
- **[MUST] 서늘하되 적대적이지 않게**: 진짜 목표·가치 축의 날카로운 질문은 *판단하려는 게 아니라 명료하게 하려는 것*이다. "왜 이런 걸 만드나"가 아니라 "이게 누구에게 왜 값진지 함께 또렷이 하자"는 태도. 답이 "내가 만들고 싶어서"여도 정당하다: 그걸 부정하지 말고, 다만 *그 사실을 명시*하게 도와 길을 잃지 않게 한다. 위축시키는 심문이 되면 진짜 답이 아니라 방어적 답이 나온다.
- **[MUST] 입문자 배려**: 사용자가 개발·기획에 익숙하지 않을 수 있다. 다음을 지킨다:
  - **쉬운 말로**: 전문 용어를 쓸 때는 괄호로 풀이를 단다 (예: "비가역 요소(되돌릴 수 없는 작업)", "멱등성(여러 번 실행해도 결과가 같은 성질)"). 용어로 겁주지 않는다.
  - **"모르면 모른다고" 안내**: 답하기 어려운 질문 뒤에는 "잘 모르겠으면 '모름'이라고 답해도 됩니다: 가정으로 채우고 넘어갈게요"를 덧붙인다. 입문자가 막혀서 멈추지 않게.
  - **입문자 신호 감지 시 자동 경량화**: 사용자가 "개발 처음이에요 / 잘 몰라요 / 비개발자예요" 같은 신호를 주면, 깊이를 한 단계 낮춰 제안하고, 용어를 더 풀어 설명하며, 한 번에 더 적게 묻는다. 부담을 먼저 덜어준다.
  - **입문자 스택 가이드 (UI 구현을 모를 때)**: "뭘로 만들지 모르겠다"는 사람이 벽 앞에서 멈추지 않게, 용도별 기본값을 추천한다. *추천일 뿐 강제가 아니다*:
    - 웹 서비스 → React + Tailwind CSS + shadcn/ui (컴포넌트 풍부, AI가 잘 다룸, 예쁜 기본값)
    - 간단한 웹/랜딩 → HTML + Tailwind CSS (가볍게 시작)
    - 풀스택 한 번에 → Next.js
    - 모바일 앱 → React Native 또는 Flutter
    - 끝에 반드시: "익숙한 게 있으면 그걸 쓰세요. 이건 2026 기준 추천이라 상황 따라 다릅니다."
    - 추천은 *대화로* 안내할 뿐, 템플릿에 특정 스택을 미리 박지 않는다(인프라 미정 원칙).
- **[SHOULD] 결정 보조**: 사용자가 답을 못 정하고 막막해하면(특히 진짜 목표·가치·대안 축), 그냥 기다리지 말고 `decision-support.md`대로 돕는다 - 선택지에 트레이드오프·가역성·추천을 붙이고, "꼭 정할 것"으로 범위를 좁힌다. 그래도 못 정하면 `[?]` 미정으로 두고 진행.

## 2. 인터뷰 축 (§0 동기 + 11축)

> 각 축은 깊이에 따라 묻거나 건너뛴다. ●=묻는다 ○=신호 있을 때만 ―=건너뜀
> 필수 3축(★)은 verify/CI가 빈칸이면 fail → 어떤 깊이든 반드시 채운다.

| # | 축 (카테고리) | 핵심 질문 | 빠른 | 표준 | 깊게 |
|---|--------------|-----------|:----:|:----:|:----:|
| 0 | **동기 (왜 만드나)** | 이걸 왜 만드나? 수익·업무효율·학습·취미·남의 요청? | ● | ● | ● |
| 1 | **★ 진짜 목표** | 요청한 수단 말고 그 뒤의 목적은? 누가 왜? | ● | ● | ● |
| 2 | **★ 범위 In/Out** | 이번에 하는 것 / 명시적으로 안 하는 것 | ● | ● | ● |
| 3 | **★ 성공 기준** | "무엇을 보이면 끝"인가? 측정·검증 가능하게 | ● | ● | ● |
| 4 | **사용자·맥락** | 누가 어떤 상황에서, 어떤 기존 시스템 위에서 | ― | ● | ● |
| 5 | **데이터·상태** | 무엇을 저장/입출력? 빈 상태·초기 상태는? | ― | ● | ● |
| 6 | **엣지·실패 모드** | 실패하면 어떻게? 부분 실행 안전? 복구 경로? | ― | ● | ● |
| 7 | **제약·위험·가역성** | 비기능 제약(성능·보안·호환), 되돌릴 수 있나 | ― | ● | ● |
| 8 | **UX 흐름** | 동선·혼란 지점·에러 화면, (UI면) 감성·첫인상·구현 스택 | ― | ○ | ● |
| 9 | **운영·관측** | 로그·모니터링·롤백, 누가 유지보수하나 | ― | ○ | ● |
| 10 | **대안·최소안** | 더 단순한 길? 안 만들어도 되는 부분?(YAGNI) | ― | ● | ● |
| 11 | **가치·검증** | 누가 왜 값을 치르나? 가장 싸게 검증할 길은? | ― | ○ | ● |

> **§0 동기는 첫 질문이자 "강도 조절기"다.** 답이 이후 축들의 무게를 정한다(아래 §2-0). 빠른 깊이에서도 한 줄로 묻는다(선택형이라 부담 없음). AI는 동기를 **판단하지 않고** 묻기만 한다 ─ 어떤 동기든 정당하다.

**깊이별 묻는 축 요약**:
- **빠른** = 0·1·2·3 (동기 + 핵심 3축)
- **표준** = 0~7 + 10 (동기 + 8축, 8·9·11은 신호 있을 때만 추가)
- **깊게** = 0~11 전부 (동기 + 11축)

## 2-0. §0 동기가 인터뷰 강도를 조절한다 (판단 아닌 조율)

> 동기를 물으면 그 답이 "완성 기준·성공 지표·수명·검증 강도"를 정한다. AI는 시장성·성공 여부를 *판단하지 않는다*(그건 환각이다). 동기를 물어 **인터뷰 무게를 맞출** 뿐이다.

**동기 선택지 (선택형, 직접 입력 가능)**:
- **수익/사업** (돈을 벌 목적)
- **업무 효율** (내/우리 일을 줄이려고)
- **학습/연습** (만들어보며 배우려고)
- **취미/재미** (그냥 만들고 싶어서)
- **남의 요청** (누가 있으면 좋겠다고 해서)
- **기타** (직접 입력)

**동기가 정하는 것 (기존 축들의 강도를 조절 ─ 새 질문을 늘리지 않는다)**:
| 동기 | 완성 기준(§3) | 성공 지표(§3) | 수명(§7·golden) | 검증·가치(§11) |
|------|---------------|----------------|------------------|------------------|
| 수익/사업 | 남이 돈 내고 쓸 만큼 높게 | 결제·매출·전환 | 오래 살 코드(유지보수·문서 제대로) | §11 강화(누가 왜 값 치르나·최소검증) |
| 업무 효율 | 내가 매일 쓸 만큼 안정 | 절감된 시간·반복 감소 | 오래 살 코드 | 실사용성 중심 |
| 학습/연습 | 돌아가면 됨(개념 증명) | 내가 배웠나 | 버려질 수 있음(테스트·문서 가볍게) | §11 생략 가능 |
| 취미/재미 | 내가 만족하면 됨 | 재미·완성감 | 버려질 수 있음 | §11 생략 가능 |
| 남의 요청 | 그 사람이 실제로 쓸 만큼 | 그 사람이 진짜 쓰나 | 상황따라 | "그 몇 명이 진짜 쓸 사람인가" 한 번 확인 |

- **[MUST] AI는 동기를 판단·평가하지 않는다.** "그건 돈 안 될 텐데" 같은 시장성 판단 금지(근거 없는 환각). 동기는 *사람의 것*이고, AI는 그에 맞춰 인터뷰 강도만 맞춘다.
- **[MUST] 어떤 동기도 정당하다.** "그냥 만들고 싶어서"도 완전한 답이다. 부정하지 말고, 그 사실을 명시해 이후 강도(가볍게)를 맞춘다.
- **[MUST] 동기가 수익/사업이면 네 가지를 이어 묻는다 (퍼널 4문 ─ 답의 질 판단은 안 하되, 답 없이는 못 넘어간다)**:
  1. **지불자**: 누가 돈을 내나? (쓰는 사람과 내는 사람이 다를 수 있다: 예. 학습자 vs 회사)
  2. **지불 대상**: 정확히 무엇에 내나? (전체 이용? 프리미엄 기능? 결과물 1건?)
  3. **발견 경로**: 첫 100명이 어디서 오나? ("몰라요"면 그 자체를 PRD에 미해결로 기록 ─ 만들어도 못 판다는 신호)
  4. **판정 기준**: 언제까지 어떤 숫자(대기자 N·결제 N·유입 N)가 안 나오면 보류/피벗하나? 다작일수록 이게 없으면 프로젝트들이 죽지도 살지도 않은 채 에너지를 나눠 먹는다.
  → 넷의 답을 PRD의 **수익 퍼널** 필드에 기록한다. 이후 /validate와 다음 추천(decision-support)은 "퍼널의 어느 단(발견→가치→결제→유지)이 막혔나"를 기준으로 삼는다.
- **남의 요청**일 때만 한 번 짚는다: "그 요청한 사람들이 실제로 쓸/값을 치를 사람인가?" (판단이 아니라 사람이 생각하게 하는 질문). 시장성 자체는 만든 후 실측으로 확인 ─ 인터뷰의 몫이 아니다.

## 3. 축별 질문 가이드 (무엇을 캐물을지)

### ★1. 진짜 목표
> 따뜻하게 물으면 따뜻한 거짓말이 나온다. 뼈를 파고드는 질문으로 진짜를 끌어낸다.
- 이 요청으로 진짜 해결하려는 문제는? 요청의 글자 말고 근본 목적.
- 이게 잘 되면 누가 무엇을 더 잘하게 되나?
- **존재 이유를 흔드는 질문** (하나씩, 답을 피하면 더 좁혀 되묻는다):
  - 이건 **누가 필요로 하는** 서비스인가? 아니면 **내가 만들고 싶어서** 하는 것인가? (둘 다 정당하나, 섞으면 길을 잃는다)
  - 이게 **세상에 없으면** 누가 곤란한가? "있으면 좋다"와 "없으면 안 된다"는 다르다: 어느 쪽인가?
  - 비슷한 게 이미 있나? 있다면 **왜 그걸 안 쓰고** 이걸 만드나?
  - 이걸 **안 만들면** 무슨 일이 생기나? "아무 일도 안 생긴다"면, 왜 만드나?

### ★2. 범위 In/Out
- 이번에 만드는 것 한 줄 / 명시적으로 *안 하는 것* (과잉구현 방지 계약).
- "나중에 할 것"과 "이번에 할 것"의 경계는?

### ★3. 성공 기준
- 무엇을 보이면 "끝"인가? 명령·테스트·출력으로 확인 가능하게.
- 모호어("잘 동작") 금지. 숫자·관찰 가능한 결과로.

### 4. 사용자·맥락 (역할이 여럿이면 역할별로 깊게)
- 누가 쓰나(역할·기술 수준)? 어떤 상황·빈도로?
- 어떤 기존 시스템·데이터·제약 위에 올라타나?
- **역할 파악 (먼저)**: 이 서비스를 쓰는 **주요 역할이 몇 개**인가? (예: 이용자+관리자 / 관리자+현장담당 / 고객+매니저+관리자 등: 구성은 프로젝트마다 다르다)
- **역할이 2개 이상이면: 각 역할마다 따로 캐묻는다**:
  - 이 역할은 **무엇을 하려고** 들어오나? (핵심 목적 1줄)
  - 이 역할의 **여정(Step)**: 처음 진입 → ... → 목적 달성까지 단계로 나누면? (예: 탐색 → 가입 → 대시보드 → 요청 → 확인 → 평가)
  - 각 Step에서 **어떤 화면을 보고, 무엇을 할 수 있나**? (화면 단위 구체성: 아래 §4-screen 참조)
- 역할이 1개(단일 사용자·자동화)면 이 분리는 건너뛴다: 과하다.

> **§4-screen 화면 구체성 (UI가 있는 서비스일 때)**: 각 Step의 화면을 모호하게 두지 않는다. 화면 하나당 최소 이 셋을 묻는다: (1) 이 화면에서 **사용자가 보는 것**(핵심 정보·카드·목록), (2) **할 수 있는 행동**(버튼·입력·선택), (3) 행동의 **결과**(다음 화면·알림·상태 변화). "대시보드 만든다"가 아니라 "대시보드엔 X·Y·Z가 뜨고, A버튼 누르면 B가 된다"까지. 빈 상태·에러 화면도 포함(§6 데이터·엣지와 연결).

### 5. 데이터·상태
- 입력은 무엇, 출력은 무엇? 어디에 저장되나?
- **빈 상태**(데이터 0개)일 때 화면·동작은? 초기값은?
- 동시에 여러 번 실행되면? 멱등성이 필요한가?

### 6. 엣지·실패 모드
- 실패하면 어떻게 되나? 부분 실행으로 데이터가 오염되지 않나?
- 외부 의존(API·DB)이 죽으면? graceful degradation 있나?
- 가장 흔한 사용자 실수 3가지와 그때 동작은?

### 7. 제약·위험·가역성
- 비기능 제약: 성능 목표·보안 요건·호환성?
- 비가역 요소(되돌릴 수 없는 작업: 예: 삭제·배포·마이그레이션·외부전송)가 있나? → 있으면 '풀' 등급.
- 잘못되면 어떻게 되돌리나(롤백 경로)?
- **"놓치면 서비스가 죽는 리스크"**가 있나?: 비용 폭주(트래픽·API 폭증), 법·규제(개인정보·민감정보 처리), 보안 사고 중 해당되는 게 있나? (실서비스·돈·개인정보 다룰 때만 깊게)
- **[MUST] 보안 프로파일 확정**: 이 프로젝트가 다음 중 하나라도 다루나? ─ 데이터 저장 / 로그인·인증 / 결제 / 개인정보 / 사용자 입력 / 외부 공개. 하나라도 해당하면 **"보안 민감" 프로젝트**로 표시하고, 이후 모든 관련 작업의 검증에서 golden-set G15-G19(보안 유형)를 필수 적용한다. 보안은 나중에 붙이는 게 아니라 착수 시 등급이 정해진다.

### 8. UX 흐름
- 처음 쓰는 사람의 동선을 한 줄씩 따라가면? 어디서 막히나?
- 에러·로딩·빈 상태 화면이 설계됐나?
- 가장 자주 쓸 동작이 가장 쉽게 닿나?
- 사용자가 이탈하거나 전환(가입·결제·완료)을 포기하는 마찰 지점은 어디인가? 그 허들을 어떻게 낮추나? (수익·전환이 중요한 서비스일 때만)

> **§8-design 감성·디자인 품질 (UI가 있는 서비스일 때만)**: "작동하는 UI"를 넘어 "갖고 싶은 UI"를 묻는다. 같은 기능이라도 첫인상·감성이 흥행을 가른다(예: 평범한 카드를 캐릭터 디자인으로 차별화해 1위).
> - 첫인상에서 사용자가 **무엇을 느껴야** 하나? (신뢰 / 귀여움 / 전문성 / 재미 / 고급 등 ─ 한 단어로)
> - 비슷한 서비스와 **시각적으로 어떻게 다를** 건가? (색감·폰트·톤의 차별점)
> - 타겟 사용자가 좋아하는 **분위기**는? (그들이 이미 쓰는 서비스의 느낌)
> 이건 자동화·스크립트·내부 도구엔 불필요하다. **사람이 보고 고르는 UI 서비스일 때만** 묻는다.

> **§8-stack UI 구현 스택 결정 (UI가 있는 서비스일 때만)**: "무엇으로 만들지"를 정하게 한다. 단 **특정 스택을 강제하지 않는다**(인프라 미정 원칙). 결정을 끌어내되 답은 프로젝트가 정한다.
> - 웹인가 앱인가? 어떤 프레임워크·UI 라이브러리로 구현할 건가?
> - **사용자가 "모르겠다 / 처음이에요"라고 하면** ─ 벽 앞에서 멈추지 않게 기본값을 추천한다 (아래 입문자 스택 가이드). "이걸로 시작할까요? 나중에 바꿔도 됩니다." 선택지를 더 보고 싶어 하면 `STACK-OPTIONS.md`(잘 알려진 스택 참고 카탈로그)를 안내한다 ─ 강요가 아니라 판단을 돕는 정보다.
> - **익숙한 스택이 있다고 하면** ─ 추천하지 않고 그걸로 간다.

> **§8-env 작업 환경 확인 (사전 확인 → 문서·설정 반영)**: 스택·환경이 정해지면 그 프로젝트의 문서·설정에 반영한다(범용 템플릿엔 박지 않되, 알게 되면 채운다).
> - 어떤 **스택·도구**를 쓰는가? → 정해지면 `.claude/settings.json` 권한에 그 도구를 동사 기준으로 추가 제안(조회=allow, 배포·삭제=deny). 기준: `.claude/PERMISSIONS.md`.
> - **여러 작업 환경(여러 PC 등)을 오가는가?** → 그렇다면 멀티 디바이스 핸드오프 흐름을 적용(`memory-context §2-1`): 떠날 때 상태 파일까지 commit+push, 받을 때 pull 후 복원.

> **§8-verify 스택 맞춤 검증 주입 (모든 작업, 스택이 정해지면)**: 명세가 끝나 기술 스택(Node·Python·GAS·정적웹 등)이 정해지면, AI는 `docs/ops/kit/recipes/`에서 그 스택의 verify 레시피를 찾아 `verify.sh`의 **시작점으로 한 번 주입**하도록 돕는다 (빈 verify.sh로 시작하지 않게). 단:
> - 주입은 **처음 한 번, 사람 승인 후.** AI가 verify.sh를 *작업 중 자동으로 계속 고치지 않는다* ─ 검증 기준은 사람이 통제한다(동적 자동수정 금지).
> - 레시피의 예시 명령은 **실제 프로젝트 명령으로 교체**해야 한다(그대로 두면 안 됨).
> - 맞는 레시피가 없으면 가장 가까운 것에서 시작하거나 직접 작성.

### 9. 운영·관측
- 무엇을 로그로 남기나? 문제를 어떻게 알아차리나?
- 누가 유지보수하나? 배포·롤백은 어떻게?
- 비용(API·인프라)이 어디서 발생하나?
- 통제 불능 상황(트래픽·비용 폭주, 어뷰징)을 어떻게 멈추나?: 비용 상한·비상 정지(kill-switch)·요청 제한 같은 안전장치가 필요한가? (실서비스·외부 트래픽 받을 때만)

### 10. 대안·최소안
- 더 단순한 길은 없나? 라이브러리·기존 기능으로 될 일인가?
- 이번에 안 만들어도 되는 부분은?(BACKLOG로)

### 11. 가치·검증 (1인·초기·수익 미검증 단계에 특히 중요)
> 돈 안 되는 일에 시간 쓰는 걸 막는 축. 풀구현 전에 "값어치"부터 서늘하게 따진다.
- 이 서비스/기능에 **누가 왜 값(돈·시간·관심)을 치르나?** 그 사람이 지금 그 문제로 실제로 불편한가?
- 이 가설을 **가장 싸게 검증**하는 길은? 풀구현 말고 **수동·반자동·목업**으로 먼저 확인되나?
- **수익(또는 핵심 가치)에 직접 닿는 부분**은 무엇인가? 그게 이번 범위 안에 있나, 아니면 곁가지부터 만들고 있나?
- 이걸 만드는 데 드는 시간/비용 대비, **돌아올 것**이 있나? 없다면 *지금* 만들 이유가 있나?
- (탈출구) 답이 "잘 모르겠다"면 그것도 정직한 답이다 → 가정으로 명시하고, **검증부터** 하는 작업으로 범위를 좁힌다.

## 4. 완료 기준: 체크리스트 + 점수 (이중 강제)

인터뷰 종료 전, 고른 깊이의 축들에 대해 **체크리스트와 점수를 동시에** 산출한다.

### 4-A. 체크리스트 (필수 충족: 게이트)

```
필수 (어떤 깊이든):
  [ ] ★1 진짜 목표: 채워짐 + 모호어 없음
  [ ] ★2 범위 In/Out: In과 Out 둘 다 명시
  [ ] ★3 성공 기준: 측정·검증 가능

깊이별 추가 (고른 깊이의 ● 축):
  [ ] 4 사용자·맥락   [ ] 5 데이터·상태   [ ] 6 엣지·실패
  [ ] 7 제약·위험     [ ] 8 UX 흐름       [ ] 9 운영·관측   [ ] 10 대안·최소안
```

필수 3개 중 하나라도 미충족 → **인터뷰 종료 불가**(실행 금지).

### 4-B. 완성도 점수 (충족도: 지표)

고른 깊이에서 묻기로 한 축 중 **충실히 답해진 축의 비율**:

```
완성도 = (충실히 답해진 축 가중합 / 고른 깊이의 총 축 수) × 100%

각 축 판정:
  충실(1.0): 구체적·검증가능하게 답해짐
  부분(0.5): 답했으나 모호하거나 가정으로 채움
  미답(0.0): 건너뜀 / "네가 정해"

예) 표준(8축)에서 6축 충실 + 2축 부분 = (6 + 1.0)/8 = 87.5%
```

**점수 해석** (강제가 아니라 신호):
- 90%+ : 완성형에 가까움 → 바로 실행 권장
- 70~89%: 실행 가능하나 부분 축은 PRD에 가정으로 명시
- 70% 미만: 깊이를 한 단계 올리거나 부분 축을 더 캐물을 것을 제안

### 4-C. 인터뷰 결과 서식 (PRD 말미에 첨부)

```markdown
## 인터뷰 완료 리포트
- 깊이: <빠른/표준/깊게>
- 체크리스트: 필수 3/3 · 추가 N/M
- 완성도 점수: <%> (충실 X · 부분 Y · 미답 Z)
- 가정으로 채운 축: <목록 또는 없음>
- 깊이 상향 제안 여부: <했음/안 함 + 이유>
```

## 5. 두 진입점

- **`/init-project`**: 빈 폴더에서 *제품 자체*를 시작. 축을 제품 수준으로(문제·KPI·페르소나·마일스톤) → `docs/specs/PRD.md` 채움. 기본 깊이 = **깊게**(제품은 완성도 중요).
- **`/spec`**: 기존 프로젝트에 *작업 하나* 추가. 축을 작업 수준으로 → `docs/plans/PLAN-<이름>.md`(prd.template 구조) 채움. 기본 깊이 = **표준**.

둘 다 시작 시 깊이를 한 번 확인하고, 사용자가 바꾸면 그대로 따른다.

## 6. 종료 조건 (이게 충족돼야 인터뷰 종료 → 실행 가능)

> 아래가 **전부** 충족되기 전에는 코드·파일 생성을 시작하지 않는다. 하나라도 빠지면 인터뷰는 아직 진행 중이다.

- **고른 깊이의 모든 축을 다 물었다.** (빠른=3 / 표준=8 / 깊게=11) 건너뛴 축은 사용자 동의 또는 명시적 "그만"으로만 허용되고, 그 사실이 리포트에 기록됐다.
- **체크리스트 필수 3축**(진짜 목표·범위·성공 기준)이 다 찼다 (verify/CI가 빈칸이면 fail).
- 고른 깊이의 ● 축이 충실 또는 부분으로 채워졌다(미답은 가정으로 명시).
- 완성도 점수와 인터뷰 리포트가 PRD 말미에 첨부됐다.
- 진행을 막던 열린 질문이 해소됐거나, 남은 건 가정으로 명시됐다.
- **사용자가 "이대로 진행" 확인했다.** ← 이 확인 없이는 절대 실행 금지.

> 자가 점검: 코드를 짜기 직전, 스스로 물어라: "모든 축을 물었나? 사용자가 진행을 확인했나?" 둘 중 하나라도 No면 멈추고 인터뷰로 돌아간다.
