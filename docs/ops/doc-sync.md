# 문서 동기화 (Doc-Sync): 코드↔문서 불일치 탐지·갱신

> **파일명 유래**: `doc`(문서) + `sync`(동기화). 코드와 문서가 어긋나지 않게 맞추는 규칙이라 이 이름이다. 낡은 문서 = 오정보라는 문제의식에서 나왔다.

> 목적: 구현이 바뀌면 명세·CLAUDE.md 등이 *낡지 않게*. 방식 = **탐지는 자동, 갱신은 사용자 승인**.
> 자동 덮어쓰기 금지(잘못된 구현이 명세를 오염시키고, 결정 의도가 사라짐). 사람이 승인한 것만 반영.

## 언제 (트리거)
- **루프 끝날 때마다**(⑤보고 직전): 이번 작업이 바꾼 것과 문서가 어긋나는지 점검한다.
- 또 작업 *도중* 불일치를 발견하면 즉시(agent-loop ⑤ 자가치유 MUST).

## 무엇을 보나 (변경 → 점검할 문서 매핑)
> 이번 변경이 아래에 해당하면, 짝 문서가 현실과 맞는지 확인하고 어긋나면 diff 제안.

| 이번에 바뀐 것 | 점검·갱신 후보 문서 |
|----------------|---------------------|
| API 엔드포인트·계약 | `docs/specs/API_SPEC.md` |
| 데이터 구조·스키마 | `docs/specs/DB_SCHEMA.md` (또는 해당 문서) |
| 아키텍처·인프라·데이터 흐름 | `docs/specs/ARCHITECTURE.md` |
| 스택·구조·컨벤션·위험작업 결정 | `CLAUDE.md` §3·§4 + 관련 `docs/specs/decisions/ADR-*` |
| 제품 목표·범위·KPI | `docs/specs/PRD.md` |
| 검증 방법·통과 기준 | `verify.sh` + `CLAUDE.md` §2 |
| 작업 완료 | 해당 `docs/plans/PLAN-*.md` 상태 Done + `docs/plans/CHANGELOG.md` 한 줄 |
| PLAN/PRD 문서 내용 수정 | 그 문서 헤더의 "최종 수정" 날짜를 오늘로 갱신(생성일·파일명은 유지) |
| PLAN 새로 만듦 / 상태 바뀜 / 완료 | `docs/plans/INDEX.md` 행 추가·상태 갱신. **PLAN을 Done으로 닫는 시점에 반드시 묻는다**: "이 PLAN, 참조도 끝났나요? `archive/`로 옮길까요?" ─ 승인 시 이동(mv, 확인 권한) + INDEX 행을 "완료" 섹션으로. 자동 이동은 안 한다(참조 중인 문서의 링크가 깨질 수 있어 사람이 판단) |
| 미해결 항목 해결됨(OQ-*) | 그 항목이 적힌 ADR/PRD의 상태 갱신 |

> 매핑에 없는 문서라도, "이 변경으로 사실과 달라진 서술"을 발견하면 동일하게 제안한다.

## 어떻게 (절차)
1. **탐지**: 위 매핑으로 "이번 변경과 어긋나는 문서·문장"을 찾는다. 없으면 "동기화 필요 없음"이라고 보고.
2. **제안**: 어긋난 곳마다 **무엇을→무엇으로 바꿀지 diff(또는 before/after)**로 제시. 추측 금지(실제 코드·문서 근거).
3. **승인 대기**: 사용자가 승인한 항목만 갱신. 보호 문서는 별도 승인.
4. **갱신**: 승인분 반영. 결정·이유가 사라지지 않게: *상태/사실*만 갱신하고 *왜*는 보존(필요하면 ADR에 추가).

## 하지 않을 것 (안전장치)
- 승인 없이 자동 덮어쓰기 금지. 특히 PRD의 "왜"·ADR의 결정 이유는 코드 상태로 덮지 않는다.
- 잘못됐을 수 있는 구현을 "정답"으로 명세에 박지 않는다: 불일치 시 *어느 쪽이 맞는지*를 사용자에게 묻는다.

## 버전 릴리스 시 (공개 레포)
버전을 올릴 때는 세 곳을 함께 갱신한다 (빠뜨리면 사용자가 변경을 모른다):
1. `CHANGELOG.md` (루트): 공개용 마일스톤 요약
2. `README.md`의 "버전 히스토리": 최근 몇 개 한 줄 요약
3. `docs/plans/CHANGELOG.md`: 작업 단위 상세 이력 (내부용)
