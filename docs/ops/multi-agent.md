# 멀티에이전트 운영 패턴 (Multi-Agent Pattern)

> **파일명 유래**: `multi-agent`(여러 에이전트). AI 여럿이 역할을 나눠 협업할 때의 패턴·경계라 이 이름이다.

> `agent-loop.md`를 보조하는 역할 분리 패턴. **멀티에이전트를 쓸 때만 적용** (단일 에이전트 작업은 건너뜀).
> 최종 검토일: 2026-06-18.

---

## 0. 적용 판단

아래 중 하나라도 해당하면 이 문서를 적용한다:
- 둘 이상의 Claude Code 인스턴스·서브에이전트가 동시에 작업
- 에이전트가 다른 에이전트를 호출하는 구조 (orchestrator → sub-agent)
- 역할이 다른 에이전트(기획 / 실행 / 검토)를 순서대로 또는 병렬로 사용

단일 에이전트 단순 작업 → **건너뜀**.

## 0-1. 멀티에이전트의 두 형태 (먼저 구분하라)

"멀티에이전트"는 AI 둘 이상이 동시에 일하는 것을 통칭하지만, 두 형태가 있고 **위험이 다르다.**

| | 형태 A: 한 세션 오케스트레이션 | 형태 B: 여러 세션 동시 |
|---|---|---|
| 구조 | 한 에이전트(지휘자)가 서브에이전트 여럿을 거느림 | 세션 창을 여러 개 열어 각각에 AI가 붙음 |
| 지휘자 | **있음** (한 에이전트가 전체를 조율) | **없음** (서로의 존재조차 모름) |
| 충돌 조율 | 지휘자가 경로·순서를 조율해 막을 수 있음 | **아무도 못 막음** (각자 "내가 전체 담당"이라 여김) |
| 이 문서의 §1~§5 | 적용됨 (역할·경로·게이트를 지휘자가 강제) | **AI 규칙만으로는 부족** → §2-1 필수 |

> **[MUST] 형태 B의 핵심 경고**: 여러 세션이 **같은 폴더·같은 파일 시스템**을 동시에 보면, 각 AI는 다른 세션의 변경을 모른 채 자기 시작 시점의 상태를 기준으로 덮어쓴다. 그 결과 한쪽 작업이 통째로 사라진다(서로의 산출물을 지움). **이건 AI에게 "남의 것 건드리지 마"라고 지시해도 못 막는다** ─ 각 세션의 AI는 다른 세션을 인지할 수단이 없기 때문이다.
>
> **형태 B의 해결은 AI 규칙이 아니라 사람이 작업 공간을 물리적으로 분리하는 것이다** (§2-1). 그래서 **기본 권장은 형태 A(한 세션에서 순차 또는 오케스트레이션)** 이고, 형태 B는 작업 공간 분리가 전제될 때만 쓴다.
>
> ⚠ "동시에 열면 빠르다"는 착각이다. 충돌로 서로 지우고 다시 만들면 순차보다 **느려진다.** 진짜 독립된 큰 작업이고 작업 공간을 분리했을 때만 형태 B가 빠르다.

## 1. 역할 3분리 원칙

> 이미지 11번 멀티에이전트 개념: 기획(Planning) / 실행(Execution) / 검토(Review) 분리.
> 역할 경계가 무너지면 에이전트가 목표를 스스로 재정의하거나 검증 없이 완료를 선언한다.

| 역할 | 담당 | 금지 |
|------|------|------|
| **기획 에이전트** | spec-interview 진행 → PRD 초안 생성 | 코드·파일 직접 생성 금지 |
| **실행 에이전트** | PRD 기반 코드·파일 생성, ./verify.sh 실행 | PRD 목표 재정의 금지, 자체 성공 기준 창작 금지 |
| **검토 에이전트** | 결과물이 PRD 수용 기준과 일치하는지 검증 | 직접 수정 금지, 통과/보류/재시도 판정만 |

### 역할 전환 규칙
- 역할은 **사람이 지정**한다. 에이전트가 자의로 역할을 바꾸지 않는다.
- 실행 에이전트가 PRD가 불명확하다고 판단하면 → 실행 중단 + 기획 에이전트 또는 사람에게 반환.
- 검토 에이전트가 "보류"를 반환하면 → 실행 에이전트가 재작업, 재검토 사이클.

## 2. 경로 분리 원칙 (동시성 안전)

병렬·다중 에이전트 작업 시 충돌 방지:

```
src/
  feature-a/   ← 에이전트 A 전담
  feature-b/   ← 에이전트 B 전담
  shared/      ← 변경 시 반드시 사람 승인 후
```

- **공유 상태**(DB·외부 API·shared/)는 lock 또는 idempotency key 필수.
- 같은 파일을 두 에이전트가 동시에 수정하지 않는다. 경합이 생기면 **먼저 멈추고 사람에게**.
- 서브에이전트는 `docs/plans/PLAN-<이름>.md`에 담당 경로를 명시하고 그 범위 밖은 건드리지 않는다.
- **[MUST] 담당 경로 밖은 "무시"한다 (보지도 마)**: 담당 경로 밖 파일은 읽기·수정·삭제·정리 **일체 금지**. 그 파일이 불필요·미완성·이상해 보여도 **존재하지 않는 것처럼 무시한다.** (다른 에이전트가 작업 중일 수 있다. "정리하려는" 본능이 남의 작업을 지운다 ─ 실제 사고 사례.) "건드리지 마"보다 강하게: *쳐다보지도 마.*

## 2-1. 작업 공간 분리 (형태 B: 여러 세션 동시일 때 필수)

> §0-1 형태 B(여러 세션)는 경로를 나눠도 충돌한다. 같은 파일 시스템을 공유하면, ① 공유 파일(설정·라우팅·index)을 서로 덮어쓰고 ② "정리 본능"으로 남의 경로를 지우기 때문이다. **AI 지시가 아니라 사람이 작업 공간 자체를 분리해야 한다.** 셋 중 하나:

| 방법 | 어떻게 | 비고 |
|------|--------|------|
| **순차 (가장 안전·권장)** | 세션 하나로 "A 끝내고 B" | 충돌 0. 대부분 이게 제일 빠르다 |
| **git 브랜치** | 세션마다 다른 브랜치(`git checkout -b task-a`) | 같은 프로젝트의 평행 우주. 나중에 사람이 merge. git 사용 시 |
| **폴더 복제** | 프로젝트를 두 벌 복사해 각자 작업 | git 없이도 가능. 바뀐 파일만 나중에 합침 |

- **[MUST] 같은 폴더 + 여러 세션 동시 수정 금지**: 작업 공간을 분리하지 않은 채 여러 세션이 같은 폴더를 동시에 수정하면 서로의 작업을 덮어쓴다. 분리가 안 됐으면 **순차로 전환**한다.
- 공유 파일(설정·라우팅·공통 모듈)은 어느 경우든 **한 곳에서만, 또는 사람이 직접** 관리한다.

## 3. 퀄리티 게이트 3단계 (Quality Gate)

> 이미지 14번 퀄리티 게이트: 통과 / 보류 / 재시도.
> 검토 에이전트(또는 검증 단계)가 아래 3가지 중 하나를 반환한다.

| 판정 | 조건 | 다음 행동 |
|------|------|-----------|
| **통과(Pass)** | `./verify.sh` exit 0 + PRD 수용 기준 충족 | ⑤ 보고 → 완료 |
| **보류(Hold)** | 기능은 동작하나 비기능(보안·성능·문서) 미달, 또는 PRD와 해석 불일치 | 사람 판단 요청 + 이유 명시 |
| **재시도(Retry)** | `./verify.sh` exit 1, 원인 특정 가능 | ↺ 원인규명 → 수정 → 재검증 |

**보류와 재시도의 차이**: 재시도는 에이전트가 혼자 고칠 수 있는 것, 보류는 사람의 판단이 필요한 것.

검토 에이전트는 보고 시 반드시 판정 + 근거 + 검증 출력을 첨부한다.

## 4. HITL 트리거 (Human-in-the-Loop)

> 이미지 15번 HITL: 중요 판단 전 사람 확인.
> 에이전트는 아래 중 하나라도 발생하면 **즉시 작업을 중단**하고 사람에게 올린다.

**자동 중단 트리거**:
- 같은 단계에서 재시도 **3회 초과**
- 예상 토큰·비용 상한 도달 (있으면 명시: `<이 프로젝트 상한 채우기>`)
- PRD 성공 기준들이 **서로 충돌**하는 상황 발견
- 보안·법적 판단이 필요한 결과 발생
- 예상 범위 밖 외부 시스템에 영향이 생길 때
- 실행 에이전트가 PRD 목표를 재정의해야만 진행 가능한 상황

**멈추는 방식**:
```
⑤ 보고 서식:
HITL 요청: [이유 1줄]
지금까지 한 것: ...
막힌 지점: ...
필요한 결정/정보: ...
→ 사람 응답 전까지 대기
```

## 5. 에스컬레이션 경로 (Escalation)

> 이미지 16번 에스컬레이션: 단정하면 안 되는 일은 전문가/사람에게.
> HITL 트리거 이후 또는 에이전트 간 합의 불가 시 사용.

**에스컬레이션 경로** (프로젝트별 채우기):
> 멀티에이전트·팀 작업이면 채운다. 1인·단일 에이전트면 아래 한 줄로 대체 가능:
> `에스컬레이션: 사람(나)에게 직접 보고: 별도 경로 불필요 [SINGLE-AGENT]`
> (`[SINGLE-AGENT]` 마커가 있으면 CI가 빈칸 검사를 건너뛴다)
```
담당: <이름 또는 팀>
채널: <Slack 채널 / 이메일 / 이슈 트래커>
방식: <DM / 이슈 생성 / PR 코멘트>
응답 SLA: <예: 영업일 1일>
```

**에스컬레이션 시 에이전트 행동**:
1. 작업 중인 PLAN-*.md 상태를 `Blocked`로 변경
2. 막힌 이유·지금까지 한 것·필요한 결정을 PLAN에 기록
3. 지정 채널에 요약 알림 (자동화 가능 → `<알림 방법 채우기>`)
4. 사람 응답 전까지 해당 작업 일체 금지

**에스컬레이션이 필요한 전형적 상황**:
- 에이전트가 보안 취약점을 발견했으나 수정 범위가 PRD 밖
- 서드파티 API 동작이 문서와 달라 설계를 바꿔야 할 때
- 비용·일정 영향이 커서 계획 재수립이 필요할 때
- 법적·규정 준수 관련 판단이 필요할 때
