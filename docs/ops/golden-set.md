# 골든셋: 의미 검증 질문 세트 (Golden Set)

> **파일명 유래**: `golden set`(골든셋). 정답이 검증된 기준 문항 모음을 뜻하는 ML 용어. '진짜 목표를 달성했나'를 의미로 검증하는 기준 질문들이라 이 이름이다.

> 이미지 13번 골든셋: 정답 있는 검증 질문.
> `verify.sh`는 기술적 통과(lint·빌드·테스트)를 검사한다. 골든셋은 **의미적 통과**를 검사한다.
> "기술적으로 돌아가는데 사람이 원하던 게 아닌" 결과를 방지.
> 최종 검토일: 2026-06-18.

---

## 사용 시점

작업 완료 보고(⑤) 전, 또는 검토 에이전트가 판정 전에 아래 질문으로 확인.
`/validate` 명령으로 실행 가능.

## 검증 라우팅 (작업 유형 판정 → 해당 질문 의무)

> 골든셋의 핵심 텐션. 모든 질문을 다 묻는 게 아니라(피로), **이 작업에 맞는 질문을 반드시** 묻는다.
> 맞는 검증만 하되, 그건 빠짐없이. "맞는 검증만" 때문에 느슨해지지 않도록 "반드시"로 조인다.

### 1단계: 작업 유형을 먼저 판정한다 (건너뛰기 금지)
검증을 시작하면 먼저 이 작업이 어느 유형인지 **명시적으로 판정**한다. 여러 유형에 걸치면 해당되는 것을 **모두** 적용한다.

| 작업 유형 | 판정 신호 | 의무 문항 |
|-----------|-----------|-----------|
| 모든 작업 | (공통) | **G1-G3 (필수)** |
| 자동화·스크립트 | 배치·크론·반복 실행 | G4-G5 |
| UI·대시보드 | 화면·사용자 인터페이스 | G6-G7 |
| 다역할 서비스 | PRD §2-1 작성됨(역할 2개+) | G6a-G6c |
| API·외부 연동 | 외부 호출·데이터 송수신 | G8-G9 |
| 멀티에이전트 | 에이전트 여럿 협업 | G10-G11 |
| 컴포넌트 연결 | A의 출력을 B가 참조 등 | G11a-G11b |
| AI·외부 생성 데이터 사용 | AI 응답·외부 API 결과를 받아 쓰는 작업 | G11c-G11d |
| 배포·커밋 | 커밋·푸시·배포(CI/CD)로 이어지는 작업 | G13, G14 |
| 버그 수정 | 결함 수정(표준+ 깊이) | G12 |

### 2단계: 판정한 유형의 문항을 빠짐없이 평가한다
- 판정한 유형의 의무 문항을 **하나도 건너뛰지 않는다.** 건너뛰려면 "이 문항은 해당 없음: <이유>"를 명시한다(침묵 스킵 금지).
- 검증 결과 표에 **"판정 유형: ___"을 맨 위에 적는다.** 그래야 무엇을 봤고 무엇을 안 봤는지 드러난다.
- 어느 유형에도 안 맞으면 G1-G3만. 단 "왜 다른 유형이 아닌지" 한 줄 근거를 남긴다.

### 왜 이게 텐션인가
질문이 유형별로 *있기만* 하면 실제론 G1-G3만 보고 끝나기 쉽다(느슨함). 유형 판정을 **의무**로 만들면 "UI 작업인데 사용자 동선(G6)을 안 봤다"가 드러나서 빠뜨릴 수 없다. 맞는 검증만, 단 반드시.

## 필수 3문 (모든 작업)

```
G1. 진짜 목표 일치
    PRD §진짜 목표와 결과물이 일치하는가?
    → Yes: 계속 / No: 보류(Hold) + 사람 확인

G2. 범위 준수
    명시된 범위 Out을 건드리지 않았는가?
    → Yes: 계속 / No: 즉시 되돌림 + 사람 확인

G3. 수용 기준 증명
    PRD §수용 기준을 숫자·출력으로 증명할 수 있는가?
    → Yes: 증거 첨부 / No: 보류(Hold)
```

셋 중 하나라도 No → 통과 불가. 임의 판단으로 "됐다"고 선언 금지.

## 선택 추가문 (작업 유형별)

**자동화·스크립트 작업**:
```
G4. 멱등성: 두 번 실행해도 같은 결과인가?
G5. 에러 처리: 실패 시 안전하게 멈추는가? (부분 실행 데이터 오염 없는가?)
```

**UI·대시보드 작업**:
```
G6. 실사용자 동선: 처음 보는 사람이 혼란 없이 목적을 달성할 수 있는가?
G7. 빈 상태·에러 상태가 설계되었는가?
G7a. 감성·첫인상 (사람이 보고 고르는 서비스일 때): 작동만 하는 게 아니라 "갖고 싶은가/다시 오고 싶은가"? 첫인상이 의도한 감성(신뢰·귀여움·전문성 등)에 맞고, 경쟁자와 시각적으로 구별되는가? (내부 도구·자동화는 면제)
```

**다역할 서비스 작업** (역할 2개 이상: PRD §2-1 작성한 경우):
```
G6a. 역할 누락: PRD §2-1의 모든 역할이 빠짐없이 구현됐는가? (예: 고객·매니저는 됐는데 관리자가 빠지지 않았나)
G6b. 여정 완결: 각 역할이 핵심 목적을 처음부터 끝까지(진입→달성) 막힘 없이 갈 수 있는가?
G6c. 화면 구체성: 각 화면의 (보는 것 / 할 수 있는 행동 / 결과)이 실제로 구현됐는가? 명세에만 있고 비어있지 않은가?
```

**API·외부 연동 작업**:
```
G8. 외부 서비스 장애 시 graceful degradation이 있는가?
G9. 민감 데이터가 로그·응답에 노출되지 않는가?
```

**멀티에이전트 작업**:
```
G10. 각 에이전트의 결과물이 경로 분리 원칙을 지켰는가?
G11. 검토 에이전트의 판정(통과/보류/재시도)이 문서화되었는가?
```

**여러 컴포넌트가 연결되는 작업** (A의 출력을 B가 참조 등):
```
G11a. 연결 지점 검증: 각 컴포넌트만 따로 보지 말고, A가 내보내는 것(이름·형태)과 B가 받는 것이 실제로 일치하는가? (예: A는 userName인데 B는 username으로 받지 않는가)
G11b. 빌드 통과 ≠ 연결 정상: 동적 언어(JS 등)면 잘못된 참조도 빌드를 통과한다. 그 연결 경로를 실제로 실행(또는 타입체크·통합테스트)해서 확인했는가? "로컬 빌드 됨"만으로 닫지 않는다.
```
> 근본 해결은 도구다: 타입 시스템(TypeScript)·통합 테스트를 verify.sh에 넣으면 이 불일치가 배포 전에 잡힌다. 이 문항은 그 맹점을 의식하게 하는 안전장치다.

**AI·외부 생성 데이터를 받아 쓰는 작업** (AI 응답·외부 API 결과 등):
```
G11c. 출력 신뢰 금지: AI나 외부가 준 데이터를 "그대로 믿고" 바로 쓰지 않는가? 받는 즉시 기대한 구조(필수 필드가 다 있나)를 검증하는 관문이 있는가?
G11d. 불완전 출력 대비: 필드가 누락·빈 값으로 와도 안전하게 처리되는가(재요청·기본값·명확한 에러)? AI 출력은 매번 다르다 ─ "개발 때 잘 나왔음"으로는 부족하다. 가끔 빠지는 케이스가 배포 후 런타임에 터진다.
```
> 근본 해결은 도구다: 받는 지점에 스키마 검증(예: 응답을 정의된 구조로 강제 검사)을 두면 불완전한 출력이 렌더링·저장 전에 걸린다. 이 문항은 "AI가 준 건 믿을 만하다"는 착각을 막는 안전장치다.

**버그 수정 작업** (표준+ 깊이: 빠른 깊이/오타성 수정은 면제):
```
G12. 재발 방지: 이 버그를 다시 잡아낼 테스트·가드가 verify.sh에 추가됐는가?
     → 테스트 프레임워크가 아직 없으면: BACKLOG에 "이 버그 유형 회귀 테스트" 기록 (강제 아님, 권장)
     → 프로토타입·버려질 코드면: "해당 없음"으로 명시하고 통과
```
> 의도: 같은 버그가 두 번 나지 않게. 단 **강제가 아니라 권장**: 깊이·프로젝트 성격에 따라 면제.
> BACKLOG 1번(자동 테스트 도입)과 충돌하지 않게: 테스트 인프라가 없으면 "도입 신호"로만 남긴다.

**배포·커밋 작업** (커밋·푸시·배포로 이어질 때):
```
G13. 로컬 ≠ 커밋 함정 점검: 커밋·배포 전 `git status`로 확인했는가?
     → 커밋하는 파일이 의존하는 것(import·참조)이 모두 함께 커밋(스테이징)되는가?
     → "커밋 안 된 변경(워킹트리)"이 빌드에 필요한 건 아닌가?
     → 로컬 빌드는 워킹트리(미커밋 포함)로 통과해도, 배포(CI)는 "커밋된 것만" 빌드한다.
     → "로컬에서 됨" ≠ "커밋된 상태에서 됨". 가능하면 커밋 후 그 상태로 한 번 더 빌드 검증.
     → 더 확실히: `git show HEAD:<파일> | grep <심볼>`로 "커밋된 결과물에" 그 심볼(import 대상)이 실제 있는지 직접 확인한다. (스테이징 여부보다 한 단계 확실 ─ 엉뚱한 파일을 스테이징한 경우까지 잡힘)
```
> 의도: 코드가 맞아도 '커밋한 것과 안 한 것이 갈라지면' 배포가 깨진다. 검증이 "내가 보는 워킹트리"를 보는 동안, 버그는 "워킹트리와 커밋의 차이"에 숨는다. `git status` 한 번이 막는다.
> 근본 해결: 커밋·푸시를 위험 작업으로 보고, 커밋 전 의존성 스테이징을 확인하는 습관(agent-loop ④).

**공개 유출 방지** (공개 레포에 push하거나 배포할 때):
```
G14. 민감정보가 커밋/배포에 섞이지 않았는가?
     → API 키·비밀번호·토큰·프라이빗 키·내부 URL·개인정보(고객 데이터)가 커밋에 없는가?
     → `git diff --staged`로 올라가는 내용을 눈으로 확인했는가?
     → 비밀은 코드가 아니라 환경변수·시크릿 매니저·.env(.gitignore됨)에 두었는가?
     → guard.sh가 커밋/푸시 시 비밀 패턴을 1차로 경고하지만, 완전하지 않다(사람이 최종 확인).
```
> 의도: 공개하는 순간 되돌릴 수 없다(누군가 이미 봤을 수 있음). 구조 노출은 위험이 아니지만(공개 레포는 원래 다 보임), *내용에 비밀이 섞이는 것*은 진짜 유출이다. "숨겨서 지키기"가 아니라 "공개돼도 안전하게".
> 근본 해결: `.gitignore`에 `.env`·시크릿 파일 등록 + 비밀은 처음부터 코드 밖에. 이미 커밋됐다면 히스토리에서 제거(git filter-repo 등)하고 그 키는 폐기·재발급.

## 골든셋 결과 서식

```markdown
## 골든셋 검증 결과
| 문항 | 판정 | 근거 |
|------|------|------|
| G1 진짜 목표 | ✅ Pass | PRD §목표: "..." → 결과: "..." 일치 |
| G2 범위 준수 | ✅ Pass | Out 항목 건드리지 않음 |
| G3 수용 기준 | ✅ Pass | verify.sh exit 0, 출력 첨부 |

최종 판정: 통과(Pass) / 보류(Hold) / 재시도(Retry)
```

## 게이트 강제 (v2.1)

골든셋은 더 이상 권고가 아니다. `golden-gate.sh`가 강제한다:

- **언제**: PLAN-*.md 상태를 `Done`으로 바꿀 때
- **무엇을**: §9 골든셋 검증 결과 표에 G1~G3 판정(Pass/Hold/Retry)이 있는지
- **없으면**: `verify.sh` exit 1 + CI 머지 차단

즉 **의미 검증 없이 작업을 닫을 수 없다.** verify.sh(기술)와 golden-gate(의미)가 짝을 이뤄
"돌아가지만 목표와 다른" 결과를 양쪽에서 막는다.
