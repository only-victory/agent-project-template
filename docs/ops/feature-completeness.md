# feature-completeness: "되는 것"과 "완성"의 차이

> **파일명 유래**: feature(기능)의 completeness(완성도). 기능이 "동작한다"와 "완성됐다" 사이의 간극: 통상 빠지는 디테일 목록이다.
> **용도**: 참조 문서(자동 로드 아님). 에이전트는 ②결정에서 채우기 제안할 때, /validate에서 의미 검증할 때 이 목록을 근거로 쓴다.
> **원칙**: 이 목록은 "다 해야 한다"가 아니다. **빠진 걸 알고 미루는 것**(BACKLOG)과 **모르고 빼먹는 것**의 차이를 만드는 게 목적이다. MVP면 대부분 미뤄도 된다: 단, 알고 미뤄라.

## 유형별 체크리스트

### 화면(페이지·뷰)이 있는 기능
- [ ] 로딩 상태 (데이터 오기 전 뭘 보여주나)
- [ ] 빈 상태 (데이터 0건일 때: 첫 사용자가 보는 화면)
- [ ] 에러 상태 (실패했을 때 사용자가 뭘 보나·재시도 가능한가)
- [ ] 모바일/좁은 화면 (깨지지 않나)

### 입력(폼)이 있는 기능
- [ ] 입력 검증 + 실패 시 어떤 메시지 (서버측 필수: G17)
- [ ] 중복 제출 방지 (버튼 연타·새로고침 재전송)
- [ ] 작성 중 이탈 (경고할 것인가·임시저장인가·그냥 버릴 것인가: 결정만이라도)

### 목록·검색이 있는 기능
- [ ] 많아지면 (페이지네이션·무한스크롤·상한)
- [ ] 0건일 때 (빈 상태와 "검색 결과 없음" 구분)
- [ ] 정렬·기본 정렬 기준

### 인증·계정이 있는 기능
- [ ] 로그아웃·세션 만료 시 동작
- [ ] 권한별로 보이는 것/안 보이는 것 (서버측 검사: G15·G16)
- [ ] 탈퇴·비밀번호 변경 등 계정 수명주기 (MVP면 보류 가능: 단 기록)

### 데이터를 쓰는(생성·수정·삭제) 기능
- [ ] 삭제 확인 + 실수 복구 (soft delete·휴지통·undo 중 뭐라도, 혹은 "없음"을 결정)
- [ ] 동시 수정 (마지막 저장이 이기나·경고하나: 결정만이라도)
- [ ] 감사 기록 (누가 언제 바꿨나: 필요한 도메인만)

### 외부 연동(API·결제·메일)이 있는 기능
- [ ] 실패·타임아웃 시 동작 (재시도·사용자 안내)
- [ ] 비용·호출량 상한 (폭주 방지)
- [ ] 테스트 환경과 실환경 분리 (실결제·실메일 사고 방지)

## 쓰는 법
- **계획 때**: 해당 유형의 목록을 훑고, 계획에 없는 항목은 "채우기 제안"으로. 채택 안 되면 BACKLOG에 "알고 미룸"으로 기록.
- **검증 때**: /validate에서 해당 유형 항목 중 "한다"고 했던 것이 실제 됐는지 확인.
- 이 목록에 없는 유형을 만나면: 그 유형의 "통상 빠지는 것"을 스스로 한 번 묻고, 반복되면 여기에 추가(회고 임계 3회 규칙).
