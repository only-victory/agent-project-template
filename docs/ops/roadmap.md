# 로드맵 · 진행 연속성 (Roadmap & Continuity)

> **파일명 유래**: `roadmap`(로드맵, 여정 지도). 프로젝트 전체를 단계로 나눠 '어디까지 왔고 어디로 가는지'를 추적해서 이 이름이다.

> 큰 프로젝트에서 "길 잃지 않고, 세션이 끊겨도 이어가고, 몰라도 굴러가게" 하는 안전장치.
> **opt-in**: 단계가 여러 개인 큰 프로젝트일 때만 쓴다. 작은 작업은 PLAN만으로 충분하다(이 문서 생략).

## 0. 계층 구조: 로드맵(부모)과 플랜(자식)

```
ROADMAP.md            큰 줄기 : 프로젝트 전체를 단계(마일스톤)로
  └ 단계 N            ←→  PLAN-*.md  작은 줄기 : 그 단계를 잘게 쪼갠 작업
```

- **로드맵 단계 1개 = 플랜 여러 개**. 플랜은 항상 머리에 부모 단계를 명시한다: `로드맵 단계: N. <이름>`.
- **상태 롤업**: 한 단계의 모든 플랜이 Done이면 → 로드맵 단계도 완료(`[x]`)로 올린다. 자식 상태가 부모로 굴러 올라간다. 사람이 매번 동기화하지 않는다.
- **단계 완료 판정은 검증 통과로만**: AI가 "다 됐다"고 말만으로 `[x]` 하지 않는다. 그 단계 플랜들이 verify/골든셋을 통과해야 인정한다(자가평가 함정 방지).

## 1. ROADMAP.md 서식

```markdown
# 로드맵: <프로젝트명>

## 단계
- [x] 1. 기반 셋업
- [>] 2. 핵심 기능 A          ← 지금 여기
- [ ] 3. 핵심 기능 B
- [?] 4. 결제 (보류: 수익모델 미정)
- [ ] 5. 배포

## 현재 위치
- 지금: 2단계 / 5단계
- 마지막 작업: <무엇을 어디까지 했나: 한 줄>
- 다음 진입점: <돌아오면 여기부터: 파일·함수·할 일>
- 진행 중 플랜: PLAN-A1(Done) · PLAN-A2(진행) · PLAN-A3(예정)

## 미정 목록 (사람 결정 대기): §3 참조
- [?] 결제 수단: 보류: 결제 단계 진입 시 재논의
- [?] DB 설계: 무지: 선택지 정리 필요

## 로드맵 변경 이력 (방향이 바뀌면 기록)
- <날짜>: 3단계를 B 대신 C로 변경: 이유: ...
```

> 진행률 %는 쓰지 않는다. "5단계 중 2단계", "플랜 3개 중 2개 Done" 같은 **셀 수 있는 사실**만. (못 믿을 추정치 금지)

## 2. 상태 기호

| 기호 | 뜻 |
|------|-----|
| `[x]` | 완료 (검증 통과) |
| `[>]` | 진행 중 (현재 위치) |
| `[ ]` | 예정 |
| `[?]` | **미정**: 아직 결정·확신이 없음 (§3에서 종류별로 다룸) |

## 3. `[?]` 미정 다루기: 사람은 완벽하지 않다

"모르겠다"는 약점이 아니라 개발에서 가장 흔한 정상 상태다. 억지로 답하게 하거나 진행을 막지 않는다.
**핵심 원칙: 모름이 진행을 막지 않는다.** 단, 그 모름이 다른 작업의 전제(의존성)가 되는 순간엔 짚어준다.

`[?]`는 세 종류다. 종류에 따라 다르게 다룬다:

### 보류 (Deferred): "아직 결정 못 했다"
- **의미**: 지금 안 정해도 진행은 되는 결정. 결정을 미루는 것.
- **예**: "결제를 Stripe로 할지 Toss로 할지 아직 모르겠다"
- **처리**: 합리적 **가정으로 두고 전진**한다("일단 미정으로 두고, 결제 단계 오면 다시 묻기"). 의존성을 기록해 두고, 그 결정이 필요해지는 시점에 다시 꺼낸다.

### 무지 (Unknown): "이게 뭔지/뭐가 맞는지 모르겠다"
- **의미**: 정보나 도움이 필요한 상태. 몰라서 못 정하는 것.
- **예**: "DB를 어떻게 설계해야 할지 모르겠다"
- **처리**: AI가 **선택지를 정리해 제시**하거나(각 트레이드오프 포함) 조사해서 사람의 결정을 돕는다. "네가 추천해줘"라고 위임하면 가역적 선택으로 진행.

### 불확실 (Uncertain): "이 방향이 맞는지 확신이 안 선다"
- **의미**: 검증이 필요한 상태. 필요/가치 자체가 의심스러움.
- **예**: "이 기능이 진짜 필요한지 모르겠다"
- **처리**: **가장 싸게 검증하는 방법**(목업·수동·작은 실험)을 먼저 제안한다(가치·검증 축과 연결). 검증 후 로드맵에 반영.

### 끝내 못 정하면
- AI가 **합리적 기본값을 제안**하고 "이걸로 갈게요, 아니면 말씀하세요".
- 그래도 미정이면 **가장 되돌리기 쉬운 선택**을 한다(가역성 우선).
- 그 선택을 ADR에 **"잠정 결정, 재검토 가능"**으로 기록한다. 완벽한 결정을 강요하지 않는다.

## 4. 이탈 감지 (옆길로 샐 때): 적절한 텐션

작업이 ROADMAP의 현재 단계에서 벗어나는 방향으로 가면, **그냥 넘어가지 않고** 멈춰 묻는다:

```
지금 작업이 로드맵 2단계(핵심 기능 A)에서 벗어나는 것 같아요. 어떻게 할까요?
  (a) 로드맵으로 복귀: 지금 작업을 멈추고 원래 단계로
  (b) 로드맵 수정: 이 방향이 더 좋으면 로드맵을 바꾸자 (변경 이력에 기록)
  (c) 그냥 진행: 일시적 곁가지, 로드맵은 그대로
```

- **AI가 임의로 로드맵을 바꾸지 않는다.** 방향 변경은 사람이 결정(HITL).
- 이게 "옆길로 새면 복귀 / 더 좋으면 로드맵 수정"을 둘 다 처리한다.

## 5. 기능 추가 시: 어디에 반영되나

"여기 이 기능 필요해"라고 하면, 그게 **큰 줄기냐 작은 줄기냐**로 반영 범위가 갈린다:

| 경우 | 판단 | 반영 |
|------|------|------|
| 현재 단계 안의 작은 작업 | 로드맵 안 건드림 | **PLAN에만** 추가 |
| 로드맵에 없던 큰 방향 | 방향 변경 | **로드맵 단계 추가 + 그 아래 PLAN** (사람 승인) |
| 애매함 | 판단 보류 | **사람에게 물음** (§4의 a/b/c처럼) |

> 자동으로 "항상 둘 다"가 아니다. 작은 건 플랜만, 방향을 바꾸는 건 둘 다. 애매하면 묻는다.

## 6. 세션 재개: `/resume`

노트북을 옮기거나, 다른 일로 멈췄다가 돌아왔을 때. 수동 인수인계(`/handoff`)에 의존하지 않고 로드맵의 "현재 위치"가 안전장치가 된다.

`/resume` 하면 AI가:
1. ROADMAP.md "현재 위치" + 그 단계의 PLAN 상태를 읽는다.
2. 브리핑: "지난번 N단계까지 했고, 마지막으로 X를 했어요. 다음은 Y부터입니다."
3. 미정 목록(`[?]`)도 함께 알림: "참고로 아직 결정 안 된 게 2개 있어요: ...".
4. 그 지점부터 이어서 진행할지 확인받는다.

> 작업 단락마다 ROADMAP "현재 위치"를 갱신한다. 그래서 갑자기 멈춰도 마지막 지점이 남는다.

> **단계 완료 시 회고 제안**: 로드맵 단계가 끝나면 `/retro`로 회고를 제안한다(retro.md). 반복 마찰을 증거로 쌓아 개선·v3 신호로 잇는다.
