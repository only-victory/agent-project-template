## 5. 수용 기준 ⛳ "무엇을 보이면 끝인가" (→ verify.sh / CLAUDE.md §2로 연결)
> **완료 계약(Goal Contract)**: "계속 일해"가 아니라 *무엇이 done·blocked·uncertain인지*를 처음에 박는다.
> 긴 세션·자동 루프가 끝없이 돌거나 "된 것 같다"로 애매하게 닫히는 걸 막는 장치.

**① 완료 상태 (done)**: 무엇이 되면 끝인가. 관찰·검증 가능하게, 모호어("잘 동작") 금지.
- <예: `POST /x` 가 201 반환 + 응답에 id 포함>
- <예: 기존 테스트 전부 통과 + 신규 케이스 N개 추가>

**② 검증 증거 (evidence)**: 그 완료를 *무엇으로* 증명하나. (각 항목이 명령/테스트/출력으로 확인 가능해야 함)
- <예: `npm test` 출력 green + 벤치 결과 첨부>

**③ 제약 (constraints)**: 무엇을 건드리면 안 되나. 어디까지만 허용되나.
- <예: 인증 흐름·결제 로직은 수정 금지, X 모듈만>

**④ 막힘·중단 조건 (blocked / uncertain)** ⛳ *끝없이 돌지 않게*: 어떻게 되면 멈추고 무엇을 보고하나.
- **멈춤 신호**: <예: 같은 검증 3회 실패 / 비가역 결정 필요 / 외부 의존 막힘>
- **멈출 때 남길 것**: 시도한 경로, 수집한 증거, 막힌 지점(blocker), 다음에 필요한 입력. (환각으로 "된 척" 금지)
- **uncertain 처리**: 확신 없는 부분은 `[?]` 미정으로 명시(보류/무지/불확실 ─ roadmap §3). 추측으로 done 선언 안 함.
