# 변경 내역 (Changelog)

> 이 템플릿은 책상에서 설계한 게 아니라, **실제로 여러 프로젝트를 지으며 부딪힌 문제로** 자라났다.
> 아래 각 버전은 "무엇을 겪고 → 어떻게 고쳤는지"의 기록이다. 기능 목록이 아니라 발견의 역사다.
> 형식은 [Keep a Changelog](https://keepachangelog.com) 기반.

## v2.29: "공개했는데 API 키가 딸려가면?"
**발견**: "구조를 노출하면 정보가 탈취되나?"라는 질문에서 출발. 파일 구조 노출 자체는 위험이 아니다(공개 레포는 원래 다 보이고, 숨겨서 지키는 건 약한 방어). 진짜 위험은 파일 "내용"에 API 키·비밀번호·개인정보가 섞여 공개되는 것이다. SocialMask처럼 실제로 공개하는 프로젝트에선 실질적 위험.
**개선**: guard.sh(2차 안전망)가 git 커밋/푸시에 비밀 패턴이 섞이면 경고하고 멈춘다(오탐 가능해 차단이 아니라 사람이 확인하게). golden-set G14로 "공개 전 git diff --staged 확인, 비밀은 환경변수로"를 점검 문항에 넣었다. "숨겨서 지키기"가 아니라 "공개돼도 안전하게"가 원칙.

## v2.28: "이런 스택들이 있어요 (참고용)"
**발견**: 스택을 고를 때 "어떤 선택지가 있는지" 한눈에 보여주는 정보가 있으면, 특히 입문자가 판단하기 좋다. 잘 알려진 기술을 누구나 참고하고 써보게 하고 싶었다.
**개선**: STACK-OPTIONS.md 참고 카탈로그 추가. 핵심은 "트렌드 자동 반영"이 아니라 "잘 알려지고 널리 쓰이는 것"(채택도가 곧 검증) 위주 ─ 검증 안 된 신기술을 쫓지 않는다. 강요가 아니라 정보 제공이고, 결정은 사람이. 한국 특화는 라벨을 붙여 지역이 다르면 갈아끼우게 했다. 자동 로드 안 해 토큰도 안 먹는다.

## v2.27: "이거 왜 만들어요?"
**발견**: 인터뷰가 "어떻게 제대로 만드나"엔 강했지만, 그 앞의 "왜 만드나 / 무엇을 만들 가치가 있나"라는 첫 단추가 약했다. 같은 제품도 수익 목적이냐 취미냐에 따라 완성 기준·검증 강도가 완전히 달라야 하는데, 그걸 안 물었다.
**개선**: 인터뷰 맨 앞에 §0 '동기' 축 추가(수익·업무효율·학습·취미·남의요청). 단, AI가 시장성을 "판단"하게 하진 않았다(그건 환각). 동기를 "묻기만" 하고, 그 답이 완성 기준·성공 지표·수명·검증 강도를 조율한다. 취미면 가볍게, 사업이면 빡세게. 새 질문은 하나만 늘고 나머지는 기존 축의 강도를 정하는 방식.

## v2.26: "범용 템플릿이어야 한다 (특정 사용자 색 빼기)"
**발견**: 권한 기본값(v2.24)에 특정 스택 도구(clasp·vercel·aws)가, 멀티 디바이스(v2.25)에 "PC 3대" 같은 특정 환경 표현이 박혀 있었다. 편의로 넣었지만, 범용 템플릿이 특정 사용자의 스택·환경을 기본값으로 알 이유가 없다(인프라 미정 위반).
**개선**: settings.json은 언어·스택 무관 보편 명령만(읽기·검사 + git + rm·sudo 차단). 특정 도구는 제거하고, "스택이 정해지면 동사 기준으로 추가"하는 가이드만 남김. 인터뷰(§8-env)에서 스택·멀티환경이 확인되면 그때 그 프로젝트 문서·설정에 반영하도록 연결. 템플릿엔 안 박되, 알게 되면 채운다.

## v2.25: "PC 3대를 오가며 작업을 이어가고 싶다"
**발견**: 자리·집·노트북을 옮겨 다니며 한 프로젝트를 이어가는데, PC가 바뀌면 이전 작업을 받아 이어가야 한다. 사용자가 이전 PC에서 commit+push, 다음 PC에서 pull은 하는데 ─ git은 "커밋된 것"만 옮기므로, 코드만 push하면 "어디까지 했나"가 안 넘어간다.
**개선**: memory-context에 멀티 디바이스 워크플로우 명시. 떠날 때 상태 파일(roadmap·PLAN·HANDOFF)도 함께 commit+push하고 로컬=원격 일치 확인, 받을 때 pull 후 새 세션이 그 파일들을 읽어 위치 복원(/resume)하는 흐름. 부품(roadmap·handoff·resume)은 이미 있었고, "여러 PC 오가기" 흐름을 한곳에 정리했다.

## v2.24: "매번 엔터 누르느라 옆에 붙어 있어야 한다"
**발견**: 작업을 시키면 bash 명령마다 승인을 요구해서, 자동으로 일하라고 시킨 의미가 없었다. ls·cat·test 같은 안전한 것까지 매번 물어 피곤했다.
**개선**: settings.json에 권한을 보수적 3단계로. 읽기·검사·빌드·테스트는 자동(allow), 상태를 바꾸는 건 확인(ask), 비가역·배포·삭제는 차단(deny). "npm run*"을 통째로 열면 "npm run deploy"가 새므로 build/test/lint만 명시 허용. 설령 위험한 게 allow에 새도 guard.sh가 2차로 막는다. 편의와 안전을 동시에.

## v2.23: "문서가 200개가 되니 뭐가 끝났는지 모르겠다"
**발견**: PLAN이 쌓여 100~200개가 되니, 한 폴더에 다 섞여서 "뭐가 완료고 뭐가 진행중인지" 눈으로 못 찾았다. 처음엔 "폴더를 draft/done으로 나누자"고 했지만 ─ 상태 바뀔 때마다 파일을 옮겨야 하고, 그러면 git 추적이 끊기고 링크가 깨진다(v2.22에서 피한 그 문제).
**개선**: C+A 절충. 진행 중 문서는 평평하게 두되(링크 보존), 전체를 INDEX.md 한 장의 표로 관리(상태·날짜·한줄 ─ 200개도 한눈에). 완료되고 참조가 끝난 것만 archive/로 치운다(완료분은 더 안 바뀌어 안전). 살아있는 작업과 끝난 작업을 분리하면서 git·링크는 지킨다.

## v2.22: "이 문서 언제 만들고 언제 고쳤지?"
**발견**: PLAN 문서가 쌓이는데 생성·수정 시점을 알기 어려웠다. 처음엔 "파일명에 날짜를 넣자"고 생각했지만 ─ 파일명에 수정일을 넣으면 고칠 때마다 이름이 바뀌어 git이 다른 파일로 인식(히스토리 끊김)하고, 그 문서를 가리키던 링크가 깨진다.
**개선**: 날짜를 파일명이 아니라 "파일 안 메타 헤더"에. 생성일은 한 번만 고정, 최종 수정일은 수정 때마다 갱신, 파일명은 그대로. agent-loop 보고 단계와 doc-sync에 "수정 시 최종 수정일 갱신"을 규칙으로 박아 빠뜨림 방지. 같은 목적(시점 추적)을 git·링크 깨짐 없이 달성.

## v2.21: "로컬에선 됐는데 배포가 깨졌다 (git 버전)"
**발견**: 커밋한 파일이 import하는 코드가 '아직 커밋 안 된 변경'에만 있었다. 로컬 빌드는 미커밋까지 포함해 통과했지만, 배포(CI)는 커밋된 것만 빌드해 실패. 검증이 "내가 보는 작업 폴더"를 보는 동안, 버그는 "작업 폴더와 커밋의 차이"에 숨어 있었다. 대화가 길어 규칙을 잊은 게 아니라, 검증이 보는 범위 밖이었던 것.
**개선**: 골든셋에 "배포·커밋" 검증(G13) 추가 ─ 커밋 전 git status로 "의존하는 게 다 함께 커밋되는가" 확인. "로컬에서 됨"을 "커밋된 상태에서 됨"으로 착각하지 않게. v2.17·v2.18의 "내 환경≠실제 환경" 가족에 git 버전을 더한 것.

## v2.20: "세션 둘이 서로의 작업을 지웠다"
**발견**: 한 프로젝트에서 세션 2개를 열어 각각 다른 교육 프로그램을 동시에 고쳤더니, 둘이 서로의 산출물을 지우는 사고. 경로를 나눠줘도 소용없었다 ─ 각 세션의 AI는 다른 세션의 존재를 몰라서, "정리하려는" 본능으로 남의 작업을 지우고 공유 파일을 덮어썼다.
**개선**: 멀티에이전트를 두 형태로 구분(한 세션 오케스트레이션 vs 여러 세션 동시). 여러 세션은 AI 규칙으로 못 막으니 사람이 작업 공간을 분리(순차·git 브랜치·폴더 복제)해야 함을 명시. "담당 경로 밖은 보지도 마"를 [MUST]로. 기본은 순차 권장 ─ "동시가 빠르다"는 충돌로 더 느려지므로 착각.

## v2.19: "끝났다고 말할 수 있는 조건"
**발견**: 코딩 에이전트 도구들(Codex·LazyCodex·Gajae-Code)이 독립적으로 같은 패턴으로 수렴한다는 분석을 봄 ─ 우리와 같은 방향(또 한 번의 외부 검증). 거기서 우리 빈틈을 발견: "무엇이 끝(done)인가"는 묻는데 "무엇이 막힘(blocked)인가"가 약했다. 그래서 긴 작업이 끝없이 돌거나 "된 것 같다"로 애매하게 닫힐 수 있었다.
**개선**: PRD 성공 기준을 완료 계약 4요소로 강화 ─ 완료 상태·검증 증거·제약·막힘 조건. 특히 막힘 조건(어떻게 멈추고 무엇을 보고하나, 확신 없으면 [?])을 명시해 "끝없이 돌거나 거짓 완료"를 막았다. 도구들의 tmux·worktree 같은 인프라는 추종하지 않고 관점만 흡수.

## v2.18: "AI가 준 데이터를 그대로 믿었다"
**발견**: AI가 생성한 분석 결과 JSON의 일부 필드가 가끔 비어서 오는데, 그걸 그대로 렌더링하다 배포 후 런타임 오류. 빌드도 개발검증도 통과했다 ─ AI 출력은 매번 달라서, 개발 중엔 우연히 잘 채워졌고 실사용에서 가끔 빠진 것. "AI가 준 건 믿을 만하다"는 착각이 원인이었다.
**개선**: 골든셋에 "AI·외부 생성 데이터 검증"(받는 즉시 구조 검증 + 불완전 출력 대비) 추가. AI 출력 쓰는 작업일 때만 켜지는 조건부. 근본책은 받는 지점 스키마 검증이라는 것도 명시. 단 특정 라이브러리는 안 박음(관점만 보편화).

## v2.17: "빌드는 됐는데 배포에서 터졌다"
**발견**: 컴포넌트 A가 내보내는 걸 B가 참조하는데 이름·형태가 안 맞아 배포 단계에서 오류. 빌드·lint는 통과해서(특히 동적 언어) 검증을 빠져나갔다. 가끔이지만 실서비스가 안 뜨니 아팠다. 그리고 em-dash 방지를 매번 손으로 청소만 했지 규칙으로 못박은 적이 없었다.
**개선**: 골든셋에 "연결 지점 검증"(A 출력 = B 입력 일치, 빌드 통과 != 연결 정상) 추가, 근본책은 타입체크·통합테스트를 verify.sh에 넣는 것임을 명시. em-dash는 guardrails에 사전 금지 규칙으로 못박아 사후 청소에서 사전 방지로 전환.

## v2.16: "외부 리뷰를 받았다, 다 받지는 않았다"
**발견**: 템플릿 리뷰 의견 5개를 받음. 좋은 것도 있고(위험 명령 인터셉터), 우리 구조를 오해한 것도 있었다(1인이니 멀티에이전트 삭제 ─ 멀티에이전트는 AI가 여럿이란 뜻이지 사람 수와 무관).
**개선**: 옥석을 가려 3개만 채택. guard.sh 인터셉터 메시지 강화(차단 시 승인 절차 안내), recipes verify 주입 가이드(스택 정해지면 1회 주입, 동적수정은 금지), 로드맵 복구 최우선화(컨텍스트 스위칭 대응). 삭제·통합 제안 2개는 근거를 들어 거절.

## v2.15: "작동하는 UI는 묻는데, 갖고 싶은 UI는 안 묻는다"
**발견**: 같은 서비스라도 폰트·색감·첫인상이 흥행을 가른다(평범한 카드를 캐릭터로 차별화해 1위 한 것처럼). 그런데 템플릿은 "혼란 없이 쓰나"만 묻고 "사랑받나"는 안 물었다. 동시에, 아무것도 모르는 사람은 "뭘로 만들지"(스택)에서 벽을 만났다.
**개선**: UI 서비스일 때만 감성·차별화 질문(첫인상·시각 차별점·타겟 분위기) + 골든셋 "갖고 싶은가" 검증. 입문자가 스택을 모르면 기본값 추천(React+Tailwind+shadcn 등), 단 강제 아님(아는 사람은 자기 걸로). 자동화·내부도구엔 안 켜짐.

## v2.14: "명령어를 못 외우겠다"
**발견**: 명령이 11개가 되니 "그거 뭐였지?"가 잦아짐. 사용자는 명령어 이름이 아니라 자기 상황으로 생각한다("버그 고치고 싶어"지 "/bugfix"가 아니다).
**개선**: 자연어 의도 연결. 평소 말로 하면 맞는 흐름으로. 단 모호하면 되묻고, 비가역 작업은 "이해한 것 되말하고 확인" 후 실행 (잘못 이해 위험을 위험도로 나눔).

## v2.13: "검증이 느슨하다 / 마찰이 휘발된다"
**발견**: 골든셋 질문이 유형별로 있는데 강제가 없어 G1-G3만 보고 끝나기 쉬웠다. 그리고 매번 같은 데서 막히는데 기록이 안 남아 휘발됐다.
**개선**: 검증 라우팅(작업 유형 판정 → 해당 문항 의무 평가). 회고(/retro)로 반복 마찰을 카운트, 3회 넘으면 구조적 문제로 격상. 모든 문서에 "파일명 유래" 추가.

## v2.12: "실행은 잘하는데 결정이 어렵다"
**발견**: 실제로 3~4개 프로젝트를 지으며 깨달음. 인터뷰·실행·검수는 잘되는데, 검수 후 "그래서 뭘 골라야 하지"에서 매번 막혔다. 결정·기획이 진짜 병목이었다.
**개선**: 결정·기획 보조 레이어. AI가 결정을 떠넘기지 않고 트레이드오프·가역성·추천(이유)을 함께 준다. 검수 결과를 "다음 결정"까지 연결. 단 최종 결정은 사람.

## v2.11: "노트북 옮기면 어디까지 했는지 잃어버린다"
**발견**: 노트북을 들고 이동하거나 다른 일로 며칠 멈췄다 오면, 어디까지 했는지 알 수가 없었다. 큰 프로젝트일수록 전체 그림(숲)을 잃었다.
**개선**: 로드맵 안전장치(opt-in). 단계 추적 + 세션 재개(/resume) + 이탈 감지. "모름"을 정식 상태[?]로(보류·무지·불확실) 다뤄 사람이 완벽하지 않아도 굴러가게.

## v2.10: "방법론은 있는데 '생존'은 안 묻는다 / 입문자엔 어렵다"
**발견**: 기능은 캐묻는데 실서비스의 생존 리스크(비용 폭주·개인정보 법규·전환 마찰)는 안 물었다. 동시에, 인터뷰 질문이 입문자에겐 막막했다.
**개선**: 상업 리스크를 기존 축에 가볍게 흡수(새 축 없이). 입문자 배려: 용어 괄호 풀이, "모르면 모름" 안내, 입문자 신호 시 자동 경량화.

## v2.9: "역할이 여럿인 서비스를 못 담는다"
**발견**: 실제 PRD(다역할 서비스)를 보니, 고객·매니저·관리자처럼 역할이 여럿이면 각 역할의 여정을 따로 명세해야 하는데 템플릿이 한 덩어리로만 물었다.
**개선**: 역할별 기능 분리 + 사용자 여정(Step) + 화면 구체성 체크. 단일 역할이면 자동 생략(YAGNI).

## v2.8: "인터뷰가 중간에 끊기고 개발로 새버린다"
**발견**: 실제로 /init-project를 돌려보니, 11축을 다 묻기로 해놓고 중간에 "충분한 것 같다"며 코드 생성으로 넘어가버렸다. 명세가 덜 끝났는데 만들기 시작한 것.
**개선**: 모든 축 완료 + 사용자 확인 전엔 코드 생성 금지. 진행 표시[N/총축수], 선택형 질문(번호+화살표+직접입력), 되돌리기 허용.

## v2.7: "질문이 물렁하다"
**발견**: 인터뷰가 "무엇을·어떻게"는 묻는데, "이게 진짜 필요한가, 누가 왜 돈 내나" 같은 존재 이유를 안 흔들었다. 따뜻하게 물으니 따뜻한 거짓말이 나왔다.
**개선**: "서늘한 질문"(누가 필요로 하나/없으면 곤란한가/왜 기존 걸 안 쓰나) + 11번째 축 가치·검증. 단 적대적이지 않게(심문 아닌 명료화).

## v2.6: "공개하려니 정비가 안 됐다"
**발견**: 공개 레포로 내려면 사전 준비물·표기 정리가 필요했다.
**개선**: PREREQUISITES.md(가져오기는 Node.js만), AI 글 티 나는 긴 줄표 제거 등 표기 정리.

## v2.5: "매번 zip 풀고 복사하기 번거롭다"
**발견**: 새 프로젝트마다 템플릿을 수동 복사하는 게 번거로웠다.
**개선**: 공개 창고 모델. `npx degit` 한 줄로 설치. install.sh, 공개용 README, .gitignore 정비.

## v2.3: "검증은 하는데 운영·재발방지가 빈다"
**발견**: verify.sh가 스택마다 달라야 하는데 시작점이 없었고, 배포 후 운영·장애 대응과 버그 재발 방지가 빠져 있었다.
**개선**: verify.sh 스택별 레시피(Node·Python·GAS·정적웹), 운영 런북(opt-in)+/incident, 버그→재발방지 골든셋 G12.

## v2.2: "6축으론 얕다"
**발견**: 인터뷰 6축으론 데이터·실패·UX·운영 같은 중요한 면이 빠졌다. 모든 작업에 다 묻기엔 무거웠다.
**개선**: 6축 → 11축 확장 + 깊이 선택(빠른/표준/깊게)으로 무게 조절 + 완성도 체크리스트.

## v2.1: "돌아가는데 목표와 다른 결과"
**발견**: 기술 검증(verify.sh)만으론 "돌아가지만 원하던 게 아닌" 결과를 못 막았다.
**개선**: 골든셋 게이트(의미 검증 없이 작업 못 닫게) + 에스컬레이션 게이트 + 셋업 체크.

## v2.0: "1인을 넘어서면 통제가 안 된다"
**발견**: 단일 작업을 넘어 여러 에이전트·긴 세션으로 가면 통제·연속성이 무너졌다.
**개선**: 멀티에이전트 패턴, HITL 트리거, 에스컬레이션 경로, 세션 연속성(메모리·인수인계), 골든셋 의미 검증.
